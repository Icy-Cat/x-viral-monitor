# 推文历史追踪面板 · 设计

## 目标

为订阅的博主长期记录其每条推文的曝光/互动指标时序数据，在一个独立面板里以可视化方式呈现单条推文的成长轨迹，让用户能一眼看出某次互动事件（转发/点赞/回复）和浏览量激增之间的因果关系。

## 用户场景

- 在 X / X Pro 时间线刷到一条推文，作者是用户长期想盯的博主 → 一键 `☆ → ★` 加入追踪。
- 此后扩展利用 X Pro 自身的定时刷新，自动记录该博主所有推文的指标变化。
- 用户随时打开历史面板（独立页面），按博主或全局浏览所有追踪推文。
- 进入某条推文的详情页，看到浏览量曲线 + 互动事件泳道，hover 事件能看到对浏览的影响。

## 范围

**包含：**
- 订阅博主的添加 / 删除（来自时间线，也可在面板内手工管理）
- 自动采样订阅博主推文的指标，记录到本地 IndexedDB
- 历史面板独立页面，左栏博主列表 + 右栏推文列表 + 详情视图
- CSV 导出
- 数据缺失段的可视化（用户离线时段）

**不包含：**
- 跨设备数据同步（订阅列表会同步，时序数据不同步）
- 自己发推的特殊处理（如有需要，用户把自己 @ 加入订阅即可）
- 推送通知 / 提醒
- 长于 48h 的持续采样（48h 后冻结数据但保留可查）

└────────┬─────────┘
         │ 写

┌──────────────────────────────┐
│   IndexedDB: viral-history   │
│   ├─ tweets (元数据)         │
│   └─ samples (时序帧)        │

新增/修改的关键文件：
- `bridge.js` / `content.js` — 在现有 GraphQL 拦截后调用采样器，无需新增网络请求
- `sampler.js`（新）— 应用采样规则，写入 IndexedDB
- `db.js`（新）— IndexedDB 封装（schema、读写、查询）
- `subscriptions.js`（新）— 订阅列表读写（chrome.storage.sync）
- 时间线推文上注入 `☆ / ★` 订阅按钮（扩展现有 inline 渲染逻辑）
- `history.html` + `history.js` + `history.css`（新）— 独立面板页面
- `popup.html` / `popup.js` — 增加"打开历史面板"按钮
- `manifest.json` — 暴露 `history.html` 为 web_accessible_resource

## 组件

### 1. 订阅管理

- 数据：`{ subscribed: ["@handle1", "@handle2", ...] }`，存于 `chrome.storage.sync`（小，自动跨设备同步）
- 入口 A：时间线推文作者名旁注入小图标 `☆`，点击切换为 `★`，再点取消
- 入口 B：历史面板左栏底部 "+ 添加博主"（粘贴 @handle 或主页 URL）
- 已订阅推文在时间线上有左侧 2px 细色条作视觉标记

### 2. 采样器

入库判定：
1. 推文 author ∈ 订阅列表
2. 推文年龄 < 48h（基于 `created_at`）
3. 满足 B（保底间隔） 或 C（事件触发） 任一

**保底间隔（自适应）：**

| 推文年龄  | 最小写入间隔 |
|-----------|--------------|
| 0 – 1h    | 1 分钟       |
| 1 – 6h    | 5 分钟       |
| 6 – 48h   | 30 分钟      |

**事件触发：** likes / retweets / replies / bookmarks 任一字段相比上一帧有正向 Δ → 立即写一帧（不受保底间隔限制）。

**首帧基线：** 第一次看到一条订阅推文时记录基线，不计为"事件"。

### 3. 时序库（IndexedDB）

数据库：`viral-history`，版本 1。

**Object store: `tweets`**
- key: `tweet_id` (string)
- 字段：`tweet_id`, `author`, `text`, `created_at`, `first_seen_at`, `last_sampled_at`, `status` (`active` | `frozen` | `deleted`)

**Object store: `samples`**
- key: 自增
- 索引：`by_tweet` on `tweet_id`，`by_tweet_ts` on `[tweet_id, ts]`
- 字段：`tweet_id`, `ts`, `impressions`, `likes`, `retweets`, `replies`, `bookmarks`, `d_likes`, `d_retweets`, `d_replies`, `d_bookmarks`

数据规模估算：单条推文 48h 周期 ≈ 60（首小时密采） + 60（中段） + 84（后段） + 事件触发若干 ≈ 200 帧上限；每帧约 100 字节 → 20 KB / 推文。100 条推文 ≈ 2 MB，IndexedDB 轻松承载。

### 4. 生命周期管理

- 推文进入 frozen 状态：`now - created_at >= 48h` 时由采样器跳过；状态字段更新为 `frozen`
- 推文 deleted：当 GraphQL 返回 404 / errors 时标记 `deleted`，最后一帧作为终态
- 用户取消订阅某博主：已记录的推文保留（数据不丢），仅停止后续采样

### 5. 历史面板（history.html）

**布局：** 顶栏 + 双栏主体。

顶栏：标题、搜索框（按推文文本搜）、设置入口、`导出 CSV`。

左栏 · 博主列表：
- 顶部固定 "★ 全部 (N)"
- 每个博主：头像、@handle、追踪推文数、本周内峰值流速
- 底部 `+ 添加博主`
- 点击博主筛选右栏；长按 → 取消追踪

右栏 · 推文卡片列表：
- 过滤栏：排序方式（流速 / 浏览 / 点赞 / 转发 / 发布时间）、时间范围（全部 / 今天 / 7 天）、状态（活跃 / 已冻结 / 全部）
- 卡片字段：作者+时间、推文摘要、当前四件套（👁 ❤ ↻ 💬）、流速等级图标 + 浏览量 sparkline
- 点击卡片 → 详情视图

### 6. 详情视图（方案 B）

```
┌──────────────────────────────────────────────────────┐
│ ← 返回    @author · 发布时间          [打开原推文]   │
├──────────────────────────────────────────────────────┤
│ 当前指标四件套 + 流速峰值 + 爆帖指数                 │
├──────────────────────────────────────────────────────┤
│ 浏览量曲线 (蓝色面积+折线)                           │
│ ─ 数据缺失段画灰色虚线                               │
├──────────────────────────────────────────────────────┤
│ ♥ 点赞泳道  · · ● ⬤ · ·                              │
│ ↻ 转发泳道  · ⬤(+10) · ·                             │
│ 💬 回复泳道 · · · ·                                  │
├──────────────────────────────────────────────────────┤
│  0h    1h    2h    3h    4h    5h    6h              │
└──────────────────────────────────────────────────────┘
```

**圆点编码：** 一次采样间隔内的新增量 Δ；半径 `r = 4 + 2·√Δ`（开方压缩量级差）。

**交互：**
- Hover 事件圆点 → 一条淡色垂直虚线穿过浏览曲线，曲线对应点高亮，tooltip 显示 `时刻 · +N 转发`
- Hover 浏览曲线 → 该时间窗内事件圆点高亮描边
- 顶部缩略图带框选缩放
- Toggle "显示流速"：浏览曲线上叠加半透明红色层 = 每分钟浏览增量
- Toggle "标注拐点"：自动算 d²impressions/dt² 极值点，画 ⭐
- 右键事件圆点 → 复制 `14:23 +10 转发` 文本
- 顶部按钮：导出该推文全部样本为 CSV

**渲染技术：** 用原生 SVG（项目已有 starchart.js 经验，不引入图表库），`<path>` 绘制曲线和缺失段虚线，`<circle>` 绘制事件，`<line>` 做关联高亮。

### 7. Popup 入口

`popup.html` 增加一个按钮："📊 打开历史面板" → `chrome.tabs.create({ url: chrome.runtime.getURL('history.html') })`。

## 数据流

**记录路径：**
1. 用户打开 X / X Pro，bridge.js 拦截 GraphQL 响应
2. 解析每条推文 → 检查 author 是否被订阅 + 是否满足 48h 范围
3. 命中则交给 sampler.js
4. sampler.js 查询 IndexedDB 中该推文最近一帧 → 判定保底间隔/事件触发
5. 满足条件则计算 Δ 字段并写入 `samples` store；首次看到时同步写 `tweets` 元数据

**展示路径：**
1. 用户从 popup 打开 history.html
2. 页面加载时一次性读取 `tweets` store + 每条推文的最新 sample → 渲染列表
3. 进入详情时按 `tweet_id` 查询 samples 全集（已索引）→ 渲染图表

## 错误与边界情况

| 场景 | 处理 |
|------|------|
| 用户离线 / 浏览器关闭 | 不采样，曲线对应段渲染为灰色虚线 |
| 订阅推文超出当前 X Pro 视野 | 不采样；用户切到博主 Profile 列即可拉回 |
| GraphQL 字段缺失（如某次响应没 bookmarks） | 该字段保留上一帧值，Δ 为 0 |
| 推文被删除 | API 报错时把 status 改为 deleted，停止采样 |
| 同一帧时间戳重复 | 以 `[tweet_id, ts]` 索引去重，后写入忽略 |
| IndexedDB 写入失败 / 配额满 | 在面板顶栏显示警告横幅，提示导出旧数据后清理 |
| 用户取消订阅 | 已记录数据保留；新刷到的推文不再采样；UI 标记博主为"已停止追踪" |
| 数据回退（API 偶发返回更小值） | 钳制 Δ ≥ 0；不写"负向事件" |

## 测试要点

- 订阅 / 取消订阅在 chrome.storage.sync 中正确持久化并跨页面更新
- 采样器在 1h / 6h / 48h 边界正确切换间隔档位
- 事件触发不被保底间隔吞掉
- IndexedDB schema 升级路径（version 1 → 未来版本）
- 详情图在数据点稀疏 / 密集 / 含缺失段时的渲染
- 大数据量（100 推文 × 200 帧）面板列表打开速度
- CSV 导出格式与 Excel / Numbers 兼容

## 配置 / 设置（最小集合）

历史面板设置面板暴露：
- 采样档位间隔（高级用户可自定义三档分钟数）
- 默认追踪上限（达到时提醒，避免无意识订阅过多）
- 数据保留策略（默认永久；可设"自动删除 30 天前 frozen 推文"）
- 一键清空所有历史

## 暂不做（YAGNI）

- 多设备时序数据同步
- 跨推文聚合分析（这条 vs 那条对比图）
- 趋势预测 / ML
- 与其他社交平台对接
